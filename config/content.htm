在 Odoo 的 PLM 模組中, 階段 (Stages), 類型 (Types) 以及標籤 (Tags) 分別代表以下內容:

階段:

階段代表產品生命週期中的不同階段或狀態. 每個產品在其生命週期中可能經歷多個階段, 例如設計、原型製作、測試、生產等. 在 ODOO PLM 模組中, 管理者可以定義這些不同的階段, 並為每個階段配置對應的工作流程、權限與其他設置. 將產品分配到不同的階段, 可以追蹤和管理產品在其生命週期中的進展.

類型:

類型指產品的類別或類型. 在 ODOO PLM 模組中, 管理者可以為產品定義不同的類型, 以便對產品進行分類和組織. 例如: 可以將產品分為硬體產品、軟體產品、服務產品等不同類型. 定義產品類型有助於管理和檢索產品, 並根據其類型套用特定的工作流程和設置.

標籤:

標籤用於標記和分類產品的關鍵詞或簡短字串. 在 ODOO PLM 模組中, 可以為產品添加標籤, 以便對產品進行搜索、過濾和組織. 例如: 可以為產品添加標籤來指示其特定特徵、用途、市場、客戶等方面的信息. 透過使用標籤, 可以快速定位和識別與特定標籤相關聯的產品, 進而提高工作效率.

針對專案一的題目: ODOO PLM 在協同產品設計上的應用 - 以鋼球平衡台機電控制系統設計為例

專案執行過程中, 將使用 Simens NX 進行零組件繪製鋼球平衡台控制系統, 接著將零組件轉為 STL 後進入 CoppeliaSim 進行場景模擬. 並使用 ZMQ Remote API Python 進行 PID 控制系統設計, 最後使用 LaTeX 進行專案文件編寫的流程, 並且各組員每週工作日誌的記錄必須整合到分組倉儲與網誌系統中.

進入 ODOO PLM 後, 可在 Configuration 處新增以下階段、類型與標籤:

階段 (Stages):

網誌整合階段: 將各學員每週在 cd2024 個人倉儲中所編寫的網誌, 整合到分組倉儲與網誌系統中

設計與繪製階段: 使用 Simens NX 進行零組件的設計與繪製

模型轉檔階段: 將設計好的零組件轉換為 STL 格式, 並分別匯入 CoppeliaSim 加入必要感測與驅動元件, 以便建立場景

模擬與控制系統設計階段: 在 CoppeliaSim 中進行零組件的模擬, 並使用 ZMQ Remote API Python 進行 PID 控制系統的設計

文件編寫與編譯階段: 使用 LaTeX 進行專案報告的編寫, 並透過 xelatex 轉換為 PDF 格式

分組簡報製作階段: 各組員提供各階段操作過程的檔案、圖片與影片, 協同完成分組簡報的製作

類型 (Types):

Markdown 網誌類型: 代表各組員所編寫的 Markdown 格式網誌

零組件類型: 代表使用 Simens NX 進行零組件設計所建立的零組件檔案

系統場景類型: 代表在 CoppeliaSim 中進行系統模擬的 .ttt 或 .xml 場景檔案

LaTeX 文件類型: 代表使用 LaTeX 編寫專案文件的 .tex 格式檔案

圖片檔案類型: 代表專案執行過程中各組員所建立的 .png、.jpg 或 .git 圖檔

影片檔案類型: 代表專案執行過程各組員所建立的 .mp4 影片檔案

其他檔案類型: 代表不在上述其他檔案類型的待分類檔案格式

標籤 (Tags):

週工作日誌標籤: 用於標記每週工作日誌的任務和文件

PID 控制系統設計標籤: 用於標記與 PID 控制系統設計相關的任務和文件

LaTeX 文件標籤: 用於標記與 LaTeX 編寫的文件相關的任務和文件

如何在網頁中加入 Disqus 留言系統:

要將 https://blog.disqus.com/ 放入個人課程網頁中的任何一頁, 必須先透過一個 Gmail 郵箱申請 disqus 論壇之後, 再決定要在那一個頁面加入留言系統.

免費帳號只能使用 Basic 方案: https://blog.disqus.com/

設置流程:

  1. 申請一個 Gmail 帳號.
  2. 登入 Gmail 帳號後, 連線至 https://blog.disqus.com/ 選擇透過 Gmail 登記 Basic 方案.
  3. 選擇要在個人的 Github Page 網址上納入 Disqus (即 https://Github帳號.github.io/cd2024), 採客製化方式插入 Javascript 的方式引入 disqus.
  4. 取得網站對應的 disqus Javascript 程式碼.
  5. 將 html 與 Javascript 程式設定碼放入特定的網頁超文件中.

Onshape

為什麼採用 Onshape?

優點:

  1. 只要電腦 GPU 顯示卡等級符合要求, 可以直接以瀏覽器開啟, 使用最新版本, 近端電腦無需任何額外套件安裝.
  2. 多人可以直接在線上協同零組件繪圖, 所完成的零組件可以直接透過 URL 分享.
  3. 設計繪圖流程具備版次管理, 使用者可以像使用 git 的版次管理系統進行零組件特徵管理.
  4. 所有檔案全部採雲端儲存, 使用者繪圖過程無需存檔.
  5. 提供 Featurescript, 使用者可以透過類似 Typescript (Javascript) 程式語言的方式建立客製化參數特徵.

缺點:

  1. 沒有網路就無法進行設計繪圖
  2. 對一般用戶只提供特徵層級的二次開發, 更進階的零組件或延伸開發, 只提供給特定商務用戶.

產品設計不是一項簡單的事, 尤其當設計的目的, 是希望建構一套可以永續且韌性較強的組成, 而非單單只想得到能應付當下需求的內容.

以 cd2024 的分組協同期末專案而言, 可以在上列 cd2023 課程中的場景加以延伸, 其中將碰到以下議題:

  1. 附加的零組件要採用何種套件製作, 當團隊選擇一項 MCAD(Mechanical Computer Aided Design, 電腦輔助設計) 套件後, 隨即便產生, 協同團隊成員是否都能使用所選擇的套件? 各成員所使用同一套件的不同版本進行零組件繪圖, 之後的協同流程會不會產生問題?
  2. 當零件在各協同成員的 MCAD 套件中進行繪圖, 是否使用相同的單位尺寸設定? 各零組件選擇不同的絕對與相對座標系統, 對於接下來在不同套件間的轉檔, 會不會產生問題, 或者讓協同設計流程更加繁瑣?
  3. 從 MCAD 將零組件轉出時, 各成員即便選擇了相同的輸出(Export)檔案格式, 不同的轉檔格式疏密度, 對於後續的零組件應用有沒有影響?
  4. 當零組件轉入機電模擬系統時, 將牽涉到輸入(Import)縮放比例, 以及各零件轉入後的絕對與相對座標選擇, 不同的選擇會不會對後續的動態模擬系統設定造成一定的影響?尤其過程中牽涉開放機構(Open-chain) 與封閉(Closed-chain)機構組件時, 選擇不同的零組件絕對與區域座標系統之後, 會不會對之後的機構連桿運動設定造成問題?
  5. 而最後, 當所需要的基本場景已經建立完成, 由於客戶或機電整合流程的配合需求, 必須改變特定零件的組立架構, 或是零組件的參數尺寸, 前面的所有步驟都需要重來一次嗎?
  6. 以下是在面對上列諸多議題時, 以 Solvespace 建立一塊籃球板, 然後正設法放置到規定位置的參考場景檔案: cd2024_footbasket_ball_spec_add_slvs_board.7z

操作過程如下:

  1. 開啟 Solvespace, 確認作圖單位為 mm, 繪製一個 100mmx200mmx300mm 的方塊零件, 準備最為籃球板.
  2. 由於 CoppeliaSim 內定的場景地面座標為 X-Y, 因此若在 Solvespace 中建立零件, 必須要確定零件的局部座標系與絕對坐標系的關係.
  3. 由於開啟 cd2023 模擬場景後, 發現所有的零件都以其自身的局域坐標系原點, 放在零件的正中央, 因此若將零件轉入 CoppeliaSim 後, 希望精確掌握零件的定位, 則在繪製零件草圖時, 必須設定好輪廓與絕對座標系統之間的關係.
  4. 另外, 由於 CoppeliaSim 4.5.1 版會將所有的轉入零件採同一個座標原點, 而非如舊版本轉入後各自以其局部座標系統進行定位.
  5. 而且還必須了解 CoppeliaSim 內建的尺寸單位為 m, 目前 Solvespace 轉出的製圖單位則是 mm, 因此中間差別了 1000 倍, 所以將作圖單位設為 mm 的 Solvespace 零件, 轉入 CoppeliaSim 若要使用其原先的作圖尺寸, 則必須縮小 0.001 倍.
  6. 之後, 還必須利用 Edit -> Shape reference frame -> Relocate to mesh center 將各零件的參考座標系統原點, 從廣域座標系原點, 轉換到零件個別的物件中心原點.
  7. 若零件轉入 CoppeliaSim 時選擇 Z 軸向上, 則籃球板零件轉入後, 將平躺在場景中, 若要轉至與現有的足球門相同方位, 則必須分別對自己的 X 軸與 Y 軸個旋轉 90 度.
  8. 最後再任選一座球門的 X, Y 座標系統原點位置(position), 逐步將 basketball board 放到正確的位置, 結果如: cd2024_footbasket_ball_spec_add_slvs_board.7z

ps -axo pid,comm,uname | grep "python3"

3. 假如自己所分派到的網路 port 遭到占用, 可以利用下列指令查出使用該埠號的使用者帳號:

使用下列 PyQt5 程式就能自帶瀏覽器開啟 cmsimde 內部啟動的編輯網站 https://localhost:9445

browser.py

import sys


from PyQt5.QtCore import QUrl
from PyQt5.QtWidgets import QApplication


from PyQt5.QtWebEngineWidgets import QWebEnginePage, QWebEngineView


class WebEnginePage(QWebEnginePage):
    def certificateError(self, error):
        error.ignoreCertificateError()
        return True


def main(args):
    app = QApplication(args)
    webview = QWebEngineView()
    page = WebEnginePage()
    webview.setPage(page)
    webview.load(QUrl("https://localhost:9445"))
    webview.show()
    sys.exit(app.exec_())


if __name__ == "__main__":
    main(sys.argv)
